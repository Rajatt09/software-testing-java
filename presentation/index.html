Block 1

Testing Foundations

What unit testing is, why it matters, and the hallmarks of a healthy test suite.

Foundation

What is a unit test?

A unit test verifies one small piece of behavior in isolation and gives fast, trustworthy feedback.

Small scope

Focuses on one unit of behavior, isolated from infrastructure like databases.

Fast feedback

Runs in milliseconds to support a tight feedback loop during development.

Deterministic

Passes or fails reliably for a real reason, never by luck.

Riddle

  • What makes a test “small”? Is a unit a class or a behavior?
Definitions

Unit vs Integration vs E2E

Unit Tests

  • In-memory execution
  • Isolated from I/O
  • Immediate feedback
  • High volume

Integration Tests

  • Talks to DB, Network, Disk
  • Slower execution
  • Validates component wiring
E2E tests string everything together. If a test hits a real database, it is not a unit test.

Concept

Separating unit tests from integration tests keeps your primary feedback loop blindingly fast.
Why It Matters

Good tests do more than detect regressions

Confidence Teams can change code and refactor with zero fear.
Documentation Tests describe expected behavior better than wikis.
Design feedback Hard-to-test code often signals high coupling.
Faster debugging Failures localize broken behavior instantly.
Test Quality

Characteristics of a Good Test

  • Readability: A teammate can understand the setup and intent in 10 seconds.
  • Determinism: No hidden time, random numbers, or environment dependency.
  • Isolation: Test A does not depend on Test B passing or running first.
  • Signal-to-Noise: When it fails, the failure points precisely to the broken logic, not a side effect.

Simple rule

If a failing test forces you to open the debugger just to understand what broke, the test is failing at its job.
Mental Model

Trust is the currency of testing

A test suite is only valuable if the team trusts it implicitly.

  • If tests are brittle, developers ignore failures.
  • If tests are slow, developers skip running them.
  • If tests are flaky, developers rerun them until they pass.
// A broken window in your suite @Test @Disabled("Fails on Mondays, Bob is looking into it") void calculate_discount() { ... }
JUnit 5 Basics

The core annotations

@Test

Marks a method as an executable test case.

@BeforeEach

Runs before each test to create a fresh, entirely isolated fixture.

@AfterEach

Runs after each test for resource cleanup (rare in pure unit tests).

Why lifecycle matters

It prevents hidden coupling where Test 2 relies on data created by Test 1.
Assertions

Choose the most precise assertion for the intent

Weak Assertions

Good for basic booleans, but hide why something failed.

assertTrue(user.getAge() == 25); // Failure: "Expected true but was false"

Strong Assertions

Explicitly show expected vs actual states.

assertEquals(25, user.getAge()); // Failure: "Expected 25 but was 24"
👨‍💻 Live Code: Demo01_BadAssertionsTest.java & Demo02_StrongAssertionsTest.java
@Test void demo_strongAssertion() { PricingEngine engine = new PricingEngine(); BigDecimal total = engine.calculateTotal(new BigDecimal("100"), 1, false); // This tells us exactly the expected vs actual value assertEquals(new BigDecimal("105.00"), total); }

Note

AssertJ (e.g. assertThat(x).isEqualTo(y)) provides even stronger, more fluent error messages.

On-the-spot Prompt

Identify the weak assertion

@Test void testOrderTotal() { Order order = new Order(); order.addItem(new Item("Laptop", 1000)); order.addItem(new Item("Mouse", 50)); // Which is better? assertTrue(order.getTotal() == 1050); // OR assertEquals(1050, order.getTotal()); }

Why does it matter when the test fails?

Takeaway

Maximize failure visibility. The assertion should tell you EXACTLY what went wrong without debugging.
Block 2

Designing Readable Tests

Structure, naming, smells, and building tests that survive refactoring.

Test Naming

Tests are executable documentation

A failing test name should tell you exactly what business requirement just broke.

Bad: Focuses on method names

  • testCalculate()
  • testNullProcess()
  • orderTest()

Good: Focuses on behaviors and outcomes

  • calculate_appliesDiscount_whenUserIsPremium
  • should_throwException_whenEmailIsBlank
  • rejectsExpiredCoupons()

Hint

If you append "when..." and "then...", you naturally describe behaviors instead of code.

Structure

Arrange, Act, Assert

Arrange

Build the data and collaborators needed for the scenario.

Act

Execute one clear operation on the system under test.

Assert

Verify the outcome that proves the behavior.

// Arrange Order order = new Order(); order.addItem(new Item("Book", 20)); // Act order.applyDiscount(0.10); // Assert assertEquals(18.0, order.getTotal());
👨‍💻 Live Code: Demo03_MessyStructureTest.java & Demo04_CleanAaaTest.java
@Test void demo_clearAaaStructure_Confirm() { // Arrange Customer c = new Customer("1", "Bob", "bob@example.com", false); Order o = new Order("O-1", c, List.of(new OrderItem("SKU", 1, new BigDecimal("10")))); // Act o.updateStatus(OrderStatus.CONFIRMED); // Assert assertEquals(OrderStatus.CONFIRMED, o.getStatus()); }
Why AAA Helps

AAA reduces cognitive load

  • Readers know where setup ends and behavior begins.
  • Debugging becomes easier because the flow is obvious.
  • It exposes tests that are doing too much.
// Smell: Multiple Act/Assert blocks order.addItem(item); assertEquals(10, order.getTotal()); <-- Assert 1 order.applyDiscount(0.1); assertEquals(9, order.getTotal()); <-- Act 2 & Assert 2

Smell to mention

If a test has multiple “Act” steps, it may be testing too many things at once. Split it up.
Test Smells

What makes a test rot over time?

  • Brittle tests: Fails when private implementation details change, even if public behavior is the same.
  • Over-specified tests: Asserts on things that don't matter (e.g. checking exactly 12 fields when only 1 changed).
  • Hidden setup: 400 lines of base class setup make it impossible to see what the current test actually needs.
  • Duplication: Copy-pasting 50 lines of JSON setup into 20 tests.

Discussion

What makes a test brittle? If I rename a private variable, should a test break?

Design Depth

Identifying the True Unit Under Test

A "unit" is not always a single class. It is a single behavior.

Class = Unit

Mocking every other class in the system just to test Class A. Leads to brittle, implementation-locked tests.

Behavior = Unit

Testing Class A and its pure domain helper Class B together, as long as they don't hit I/O.

👨‍💻 Live Code: Demo14_TrueUnitTest.java
// DEMO: A "Unit" is a BEHAVIOR, not a CLASS. // Here we are testing Order totaling. Order heavily relies on OrderItem and PricingEngine. // Notice we do NOT mock OrderItem. We do NOT mock PricingEngine (if it were injected). // We instantiate real objects because they are pure domain logic without side-effects (I/O). // If we mocked OrderItem, the test would be incredibly brittle and unreadable. @Test void shouldCalculateTotalUsingRealSubUnits() { // Arrange Customer c = new Customer("1", "Alice", "alice@test.com", true); OrderItem item1 = new OrderItem("LAPTOP", 1, new BigDecimal("1000.00")); OrderItem item2 = new OrderItem("MOUSE", 2, new BigDecimal("50.00")); Order order = new Order("O-123", c, List.of(item1, item2)); // Act BigDecimal total = order.totalAmount(); // Assert // We know exactly what the domain objects yield. assertEquals(new BigDecimal("1100.00"), total); }

Insight

Over-isolating classes makes refactoring impossible because tests enforce class boundaries instead of behaviors.
Test Data Setup

Fixture Discipline: Builders and Factories

Don't clutter tests with irrelevant fields. Use Test Data Builders.

// Bad: Irrelevant details pollute the test User user = new User("John", "Doe", "j@j.com", "123 Main", "NY", "10001", true); // Good: Highlight only what matters for THIS test User user = TestUserBuilder.aUser() .withAdminPrivileges(true) .build();
👨‍💻 Live Code: Demo15_TestBuilderTest.java
// DEMO: Fixture discipline. // Instead of copying 10 lines of setup into every single test... @Test void testWithBuilder() { // Arrange Order order = TestOrderBuilder.aValidOrder() .withCustomer(new Customer("1", "Bob", "b@b.com", true)) .withItem("CPU", 1, new BigDecimal("500")) .build(); // Act & Assert assertEquals(new BigDecimal("500"), order.totalAmount()); } // Usually builders live in a dedicated class, but we inline it here for the demo. public static class TestOrderBuilder { private Customer customer = new Customer("Default", "Def", "def@test.com", false); private List<OrderItem> items = List.of(new OrderItem("DEFAULT", 1, BigDecimal.TEN)); public static TestOrderBuilder aValidOrder() { return new TestOrderBuilder(); } public TestOrderBuilder withCustomer(Customer c) { this.customer = c; return this; } public TestOrderBuilder withItem(String sku, int q, BigDecimal price) { this.items = List.of(new OrderItem(sku, q, price)); return this; } public Order build() { return new Order("O-TEST", customer, items); } }

Pattern

The Object Mother or Builder pattern keeps tests scoped purely to the Arrange data that impacts the Act phase.

Verification

State vs Interaction Testing

State Verification

Checking the final value of an object. Highly robust. Preferred.

assertEquals(CONFIRMED, order.getStatus());

Interaction Verification

Checking that a method was called. Can be brittle.

verify(emailService).send(orderId);
👨‍💻 Live Code: Demo08_StateWithFakeTest.java
@Test void demo_fakeStateVerification() { InMemoryInventoryService fake = new InMemoryInventoryService(); fake.reserve("BOOK", 1); // Verify STATE, not interactions. Highly robust to refactoring. assertTrue(fake.isAvailable("BOOK", 0)); // No more books left }

Rule of thumb

Always prefer state verification over interaction verification when both are possible.
Edge Cases

Negative-path & Boundary Thinking

Tests aren't just for the Happy Path. Attack your own logic.

  • Null/Empty: What if the list is empty? Strings are blank?
  • Boundaries: What happens at age 17? age 18? age 19?
  • Errors: Does the system throw an exception, or fail silently by returning null?
@Test void should_throwException_whenAgeIsBelow18() { assertThrows(IllegalArgumentException.class, () -> { service.registerAccount(17); }); }
👨‍💻 Live Code: Demo16_BoundaryTest.java
// DEMO: Negative Path & Boundaries // The spec says: "Premium members get $10 off if subtotal is strictly > $100." // We must test exactly $100.00 and exactly $100.01. @Test void shouldNotApplyDiscountRightOnTheBoundary() { PricingEngine engine = new PricingEngine(); // Exact boundary: 100.00 BigDecimal total = engine.calculateTotal(new BigDecimal("100.00"), 1, true); // No discount applied because it is not STRICTLY greater than 100 assertEquals(new BigDecimal("100.00"), total); } @Test void shouldApplyDiscountJustAboveTheBoundary() { PricingEngine engine = new PricingEngine(); // Just over boundary: 100.01 BigDecimal total = engine.calculateTotal(new BigDecimal("100.01"), 1, true); // Discount of 10.00 applied assertEquals(new BigDecimal("90.01"), total); }

Block 3

Test Doubles and Mockito

Stubs, fakes, mocks, and how not to shoot yourself in the foot with Mockito.

Taxonomy

Test doubles help isolate behavior

A test double is a stand-in for a real collaborator. Do not call everything a mock.

Stub

Returns hardcoded, pre-defined values.

Fake

Small working implementation (e.g. an InMemoryRepository).

Mock

Used specifically to verify interactions (e.g. 'was sendEmail() called?').

Spy

Wraps a real object, delegates calls, but observes them.

The distinction matters because each double solves a vastly different design problem.
Deep Dive

Fake vs Mock tradeoff

Fake (InMemory Implementation)

  • Tracks state internally
  • Great for data repositories
  • Write once, use everywhere
  • Refactoring-proof

Mock (Mockito)

  • Checks how a method was called
  • Best at service boundaries (emails/events)
  • Unique setup in every test
  • Often brittle to refactoring
👨‍💻 Live Code: Demo08_StateWithFakeTest.java
@Test void demo_fakeStateVerification() { InMemoryInventoryService fake = new InMemoryInventoryService(); fake.reserve("BOOK", 1); // Verify STATE, not interactions. Highly robust to refactoring. assertTrue(fake.isAvailable("BOOK", 0)); // No more books left }
Mockito

Three core operations

mock()

Create the dummy collaborator.

when().thenReturn()

Stub the response needed for this scenario.

verify()

Confirm the important interaction happened.

// Arrange when(inventory.isAvailable("item", 2)).thenReturn(true); // Act boolean success = payment.processPayment(request); // Assert verify(payment).processPayment(request);
👨‍💻 Live Code: Demo06_MockitoInteractionTest.java
@Test void demo_verifySystemBoundary() { EmailClient mockClient = mock(EmailClient.class); // Act mockClient.send("test@example.com", "Hello"); // Assert: We care about the interaction occurring, not the return value verify(mockClient).send("test@example.com", "Hello"); }
  • Mock collaborators, not value objects.
  • Verify important calls, not every call.
Mockito Anti-Patterns

When mocking hurts

  • Mocking Value Objects: Never mock User or List. Just instantiate them.
  • Over-verification: verify(log).info("started") makes tests extremely brittle.
  • Mocking everything: If a test has 6 mocks and 2 lines of Act code, you are testing Mockito, not your domain.
  • Return a mock from a mock: when(a.getB()).thenReturn(mockB) is a sign of severe Law of Demeter violations.
👨‍💻 Live Code: Demo07_MockitoAntiPatternTest.java
@Test void demo_mockingValueObjectsIsBrittle() { Customer mockCustomer = mock(Customer.class); when(mockCustomer.getName()).thenReturn("Bob"); when(mockCustomer.getEmail()).thenReturn("bob@example.com"); when(mockCustomer.isPremiumMember()).thenReturn(true); // The above is painful. Always do this instead: // Customer realCustomer = new Customer("1", "Bob", "bob@example.com", true); }

Anti-pattern

Over-mocking makes tests describe the IMPLEMENTATION instead of the BEHAVIOR.
Live Decision prompt

Choose the right double

Scenario: You are testing a UserService that creates a user. It needs to save the user to a database, and send a welcome email.

  • What double should you use for the UserRepository?
  • What double should you use for the EmailGateway?

Answers

  • Repo: Fake (InMemory list) to check state.
  • Email: Mock to verify the system boundary crossed.
Block 4

Workflow, Async, and Strategy

Test Driven Development, Behavior Driven Development, taming flaky Async tests, and overall strategy.

Test Driven Development

Write the test first, then earn the implementation

Red Start with a failing test. It must compile but fail.
Green Implement the absolute minimal change to make it pass.
Refactor Clean the code's design while the green test protects you.
Repeat Grow behavior in tiny, incremental steps.

Critical point

TDD is not "write tests after coding." The failing test comes first to guide the design.
Why TDD Helps

What TDD gives you

  • Focus on the API from the consumer's perspective before building internals.
  • Prevents speculative, over-engineered code ("YAGNI").
  • Results in 100% natural coverage built-in.
  • Creates natural pressure toward testable, loosely coupled design.

Common mistake

Writing five failing tests and then coding everything at once is usually too big a step.
Behavior Driven Development

BDD makes tests read like specifications

  • Focuses on shared understanding between business and developers.
  • Changes AAA to Given / When / Then.
  • Uses domain language, omitting technical jargon when possible.
@Test void givenAvailableStock_whenOrderPlaced_thenInventoryDeducted() { // Given inventory.add("Laptop", 10); // When orderService.purchase("Laptop", 1); // Then assertEquals(9, inventory.getStock("Laptop")); }
👨‍💻 Live Code: Demo17_BddStyleTest.java
// DEMO: BDD Style Mockito // Regular Mockito uses `when(...).thenReturn(...)` and `verify(...)`. // This breaks the Given/When/Then naming rhythm because 'when' acts as 'given'. // BDDMockito fixes this vocabulary clash. @Test void givenClient_whenSending_thenMessageDelivered() { // Given (Setup) EmailClient mockClient = mock(EmailClient.class); // Note the use of `given` instead of `when` // given(mockClient.isConnected()).willReturn(true); // When (Action) mockClient.send("test@example.com", "Alert"); // Then (Assertion) // Note the use of `then` instead of `verify` then(mockClient).should().send("test@example.com", "Alert"); }
BDD is mostly about communication and naming discipline, not about external tooling.
Async Testing

Async tests fail when time is guessed

Flaky style

  • Uses Thread.sleep(1000)
  • Assumes timing will always be fast enough
  • Passes locally, fails on slow CI servers

Deterministic style

  • Uses CountDownLatch, CompletableFuture, or Awaitility
  • Waits explicitly for a signal
  • Fails fast using timeouts
👨‍💻 Live Code: Demo12_FlakyAsyncTest.java & Demo13_StableAsyncTest.java
@Test void demo_stableWithLatch() throws InterruptedException { FlakyDataFetcher fetcher = new FlakyDataFetcher(); CountDownLatch latch = new CountDownLatch(1); fetcher.fetchDataAsync(latch::countDown); // Implicitly waits UP TO 1 second, but fails precisely if exceeded. boolean completed = latch.await(1, TimeUnit.SECONDS); assertTrue(completed); }

Strong takeaway

A flaky test trains teams to ignore failures. Never tolerate Thread.sleep in a test suite.
Prompt

Diagnose the Flaky Test

@Test void sendsAsyncNotification() throws InterruptedException { notificationService.triggerAsyncAlert("System Down!"); // Test fails 1 out of 10 times on CI server Thread.sleep(500); verify(emailClient).send("System Down!"); }

How would we stabilize this?

Solution

Pass down a synchronous ExecutorService in tests, or use Awaitility: await().untilAsserted(() -> verify(...)).

Strategy

The test pyramid is about balance

Unit tests Many, fast, focused, cheap to maintain.
Integration tests Fewer, boundary-focused, moderately fast.
End-to-end tests Fewest, expensive, brittle, high-value user journeys.

Healthy testing is not "100% mocked unit tests everywhere." It is the right mix of tests providing maximum diagnostic power.
Refactoring Strategy

Characterization Tests

How do you test legacy, highly-coupled code without tests?

  1. Characterize: Write coarse tests asserting current behavior (even if messy).
  2. Introduce Seams: Extract interfaces or use dependency injection.
  3. Isolate: Inject fakes/mocks into the seams.
  4. Refactor safely: Rewrite internals with the net in place.
👨‍💻 Live Code: Demo18_CharacterizationTest.java
// DEMO: Characterization Tests // A characterization test documents what the code CURRENTLY does, // even if it's messy or ugly. It's a safety net for refactoring. @Test void characterization_shouldThrowBecauseOfHiddenStaticDatabase() { // We know LegacyCheckoutManager explodes because of GlobalDatabaseConfig // Instead of fixing it first, we WRITE A TEST proving it explodes. java.lang.NoClassDefFoundError error = null; try { // new LegacyCheckoutManager().checkout(...); // This would normally throw. We pin the failure exact type. } catch (java.lang.NoClassDefFoundError e) { error = e; } // By pinning this behavior, when we extract the database config into an injected interface, // this test failing tells us we successfully broke the hidden dependency. }

Insight

A characterization test says "I don't care what this should do, I care what it DOES right now."
